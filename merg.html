🔀 Guía Definitiva: Merge y Pull Requests

En el mundo del desarrollo de software, las ramas (branches) te permiten trabajar de forma aislada. Descubre cómo unir ese trabajo al proyecto principal de forma segura.

Esta sección establece las bases fundamentales. Entender la diferencia entre la acción técnica de unir código y el proceso social de revisión es clave para cualquier desarrollador.

🌿 ¿Qué es un Merge?

Un Merge (Fusión) es un comando de Git que une dos historiales de desarrollo independientes en uno solo. Toma los commits de tu rama secundaria y los inyecta en la principal.

🛣️ La Analogía:

Imagina que el proyecto es una autopista principal (main). Creas un desvío (rama feature) para construir una gasolinera sin parar el tráfico. El Merge es conectar tu desvío de vuelta a la autopista.

☁️ ¿Qué es un Pull Request?

Un PR no es un comando de Git. Es una funcionalidad de plataformas como GitHub o GitLab. Es una solicitud formal para que tu equipo revise y apruebe tu código antes de hacer Merge.

  • 🧐 Code Review: Permite detectar errores y sugerir mejoras.
  • 🛡️ Seguridad: Evita subir código roto directamente a producción.
  • 💬 Discusión: Espacio para comentar sobre los cambios.

💻 Cómo hacer un Merge local (por Consola)

Aunque en equipos grandes se suelen usar Pull Requests en plataformas web, es fundamental saber cómo fusionar ramas directamente desde tu terminal utilizando comandos de Git. Esto es ideal para proyectos personales, para mantener ramas actualizadas o flujos de trabajo directos.

El Proceso Paso a Paso

  1. Ubícate en la rama receptora:

    Primero debes "pararte" en la rama que va a recibir los cambios. Normalmente esta es main o develop.

  2. Actualiza tu rama local (Recomendado):

    Asegúrate de descargar la última versión del servidor antes de fusionar para evitar problemas de sincronización.

  3. Ejecuta el comando Merge:

    Estando en la rama receptora, ejecuta la fusión indicando el nombre de la rama que tiene tu nuevo trabajo.

  4. Sube y limpia (Opcional):

    Envía tu rama principal actualizada al servidor y elimina tu rama de desarrollo localmente para mantener orden.

Terminal (Ejemplo Práctico)
# 1. Cambiamos a la rama principal receptora
$ git checkout main
Switched to branch 'main'
# 2. Descargamos lo último de la nube (por si acaso)
$ git pull origin main
Already up to date.
# 3. Traemos el código de nuestra otra rama hacia main
$ git merge feature/login
Updating 34b1a2c..f19e28d
Fast-forward
index.html | 15 +++++++++++++--
1 file changed, 13 insertions(+), 2 deletions(-)
# 4. Enviamos a la nube y limpiamos localmente
$ git push origin main
$ git branch -d feature/login

💡 Tip: "Fast-Forward" vs "Merge Commit"

Si nadie ha hecho cambios en main mientras trabajabas en tu rama, Git hará un Fast-Forward (como se ve en la terminal: simplemente adelanta el puntero en el tiempo de manera lineal). Si hubo cambios en ambas ramas, Git creará automáticamente un Merge Commit para unir las dos historias de forma segura. ¡El comando a usar es el mismo para ambos casos!

⚠️ ¿Y si hay Conflictos?

A veces, al intentar hacer el Pull Request, GitHub te dirá "Can't automatically merge". Esto ocurre cuando tú y otro compañero modificaron exactamente la misma línea del mismo archivo.

Git no sabe qué versión es la correcta, así que pausa el proceso y te pide a ti (el humano) que resuelvas el conflicto.

index.html - Resolviendo el conflicto
<div class="header">
<<<<<<< HEAD (Lo que hay en main)
  <h1>Bienvenido a nuestro Proyecto</h1>
======= (El divisor)
  <h1>Hola, Usuario. Bienvenido.</h1>
>>>>>>> feature/nueva-seccion (Tus cambios)
</div>

Pasos para resolverlo:

  1. Abre el archivo conflictivo en tu editor de código.
  2. Elige qué código se queda (el tuyo, el de tu compañero, o una mezcla de ambos).
  3. Borra las marcas de conflicto generadas por Git (<<<<<<<, =======, >>>>>>>).
  4. Guarda el archivo.
  5. Ejecuta git add . y git commit para confirmar la resolución.
  6. Completa el Merge.
